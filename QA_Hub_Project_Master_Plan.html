6. Phase-by-Phase Development Plan

ลำดับพัฒนาหลัก พร้อม Entry, Deliverable และ Exit Gate

Phase 0 — Discovery & Alignment 2 สัปดาห์

  • รวบรวม Pain Point และ Workflow ปัจจุบัน
  • กำหนด Product Pilot
  • กำหนด Scope MVP/Phase 2
  • กำหนด KPI และ Success Criteria
  • ยืนยัน Integration และ Security Constraint

Exit Gate

Project Charter, Scope, Backlog, Architecture Direction และ Owner ได้รับอนุมัติ

Phase 1 — Platform Foundation 3 สัปดาห์

  • Frontend/Backend Project Setup
  • Database Migration
  • Authentication และ RBAC
  • Product/Module/Environment
  • Audit Log และ Error Handling
  • CI/CD และ Dev/Test Environment

Exit Gate

Login, Permission, Product Configuration และ Deployment Pipeline ผ่าน Integration Test

Phase 2 — Requirement Management 3 สัปดาห์

  • CRUD Requirement
  • Acceptance Criteria
  • Version และ Approval
  • Attachment/Comment
  • Filter/Search/Saved View
  • Requirement Status Workflow

Exit Gate

Requirement 1 รายการสามารถ Review, Approve และ Trace ได้ครบ

Phase 3 — Test Design Repository 4 สัปดาห์

  • Test Scenario และ Test Case
  • Step Editor และ Test Data
  • Version, Clone และ Template
  • Tag/Priority/Test Type
  • Import/Export Excel
  • Requirement Coverage
  • Review/Approve Workflow

Exit Gate

Test Case พร้อม Version และ Link Requirement สามารถนำไปใช้ใน Cycle ได้

Phase 4 — Test Cycle & Execution 4 สัปดาห์

  • Cycle/Suite/Assignment
  • Execution Result
  • Actual Result และ Evidence
  • Blocked/Skipped Reason
  • Re-run/Retest
  • Execution History
  • Bulk Update และ Progress

Exit Gate

QA สามารถสร้าง Cycle, Run, Fail และแนบหลักฐานได้ครบ End-to-End

Phase 5 — Bug Management 3 สัปดาห์

  • Create Bug from Failed Test
  • Severity/Priority/Aging
  • Kanban Workflow
  • Assign Dev และ Fix Build
  • Retest และ Closure
  • Duplicate และ Related Bug

Exit Gate

Fail → Bug → Fix Build → Retest → Closed ทำงานครบและมี Audit

Phase 6 — Build, Release & Quality Gate 3 สัปดาห์

  • Build/Version/Environment
  • Release Candidate
  • Release Checklist
  • Risk และ Readiness Score
  • QA Sign-off
  • Release Note และ Rollback Plan

Exit Gate

ระบบบล็อก Sign-off เมื่อ Critical Gate ไม่ผ่าน และแสดงเหตุผลได้

Phase 7 — Dashboard & Reporting 3 สัปดาห์

  • Pass/Fail/Blocked Trend
  • Coverage และ Requirement Status
  • Bug Aging และ Severity
  • Release Readiness
  • QA Workload
  • Export Report

Exit Gate

Dashboard ใช้ข้อมูลจริงและผลรวมตรงกับ Transaction Data

Phase 8 — QA Toolbox 4 สัปดาห์

  • SQL Library แบบ Read-only First
  • API Tester
  • Data Generator
  • File Compare
  • QR/Barcode
  • Log Analyzer
  • Execution Audit และ Permission

Exit Gate

เครื่องมือทุกตัวผ่าน Security Review และไม่อนุญาต Dangerous Action โดยไม่มี Control

Phase 9 — Integration 3 สัปดาห์

  • GitHub/GitLab Repository
  • Email/Chat Notification
  • Webhook
  • Google Sheet Import
  • CI/CD Build Status
  • Automation Result Import

Exit Gate

Integration มี Retry, Error Log, Secret Storage และ Owner ชัดเจน

Phase 10 — AI Requirement & Error Assistant 4 สัปดาห์

  • Requirement → Scenario/Test Case
  • Error/Log → Root Cause Draft
  • Bug Draft และ Regression Test
  • Confidence และ Assumption
  • Human Review
  • Prompt Audit และ Data Masking

Exit Gate

AI ผลิต Draft ได้ แต่ไม่ Publish/Close/Execute อัตโนมัติ และผ่าน QA Review 100%

Phase 11 — Markdown Documentation Intelligence 4 สัปดาห์

  • Markdown Upload/Repository Sync
  • Parser: Heading, Table, List, Code Block, Front Matter
  • Business Rule Extraction
  • Source File/Section/Line Mapping
  • Generate Test Case from Documentation
  • Missing/Conflict Detection

Exit Gate

Test Case ทุกเคสมี Source Traceability และ Commit Hash ตรวจสอบย้อนหลังได้

Phase 12 — Diff, Impact & Smart Regression 4 สัปดาห์

  • Document Version Diff
  • Added/Modified/Removed Rule
  • Impacted Test Case Mapping
  • Suggested New/Updated Test
  • Regression Scope Recommendation
  • QA Approval และ Version Update

Exit Gate

Change Impact มี Accuracy ผ่าน Pilot และไม่แก้ Test Case จริงก่อนอนุมัติ

Phase 13 — Production Rollout & Optimization ต่อเนื่อง

  • Pilot ProMaxx/MyMaxx2
  • Training และ User Guide
  • Monitoring/Alert
  • Backup/Restore Drill
  • Support Model และ SLA
  • Multi-product Rollout
  • KPI Review และ Optimization

Exit Gate

Production Stable, Adoption ผ่านเกณฑ์ และมี Backlog Phase ถัดไป

9. Critical Dependencies

จุดที่ต้องควบคุมเพื่อไม่ให้ Timeline ล่าช้า

Data Model Dependency

Requirement, Test Case, Run, Bug และ Release ต้องออกแบบ Traceability ก่อนพัฒนา UI เชิงลึก

Security Dependency

RBAC, Audit และ Secret Management ต้องพร้อมก่อน Toolbox, Repository และ AI Integration

AI Dependency

AI ต้องรอ Schema, Template และ Review Workflow เสถียร ไม่ควรเริ่มจาก Prompt อย่างเดียว

Git Dependency

Markdown AI ต้องใช้ Repository Sync, Commit และ File Version ที่เชื่อถือได้

Reporting Dependency

Dashboard ต้องอิง Transaction จริงหลัง Requirement/Test/Bug พร้อมแล้ว

Adoption Dependency

ต้องเตรียม Test Case Template, Import Data และ Training ก่อน Pilot Go-live

12. Project & Product KPI

ใช้ติดตามทั้งการส่งมอบโครงการและผลลัพธ์ทาง QA

Project Delivery KPI

  • Sprint Goal Achievement ≥ 85%
  • Critical Milestone Delay ≤ 10%
  • Escaped Critical Defect = 0
  • Rework จาก Requirement ไม่ชัดเจน ≤ 10%
  • UAT Acceptance ≥ 90%

Operational QA KPI

  • Requirement Coverage ≥ 95%
  • Test Case Reuse เพิ่มขึ้น ≥ 30%
  • เวลาเตรียม Test Case ลดลง ≥ 40%
  • Bug Traceability ≥ 95%
  • Release Readiness Report ใช้งานทุก Release

AI KPI

  • AI Draft Acceptance Rate ≥ 70%
  • Source Traceability = 100%
  • Critical Hallucination = 0
  • เวลา Review ต่อ Draft ลดลงต่อเนื่อง
  • ทุก AI Output มี Reviewer

Adoption KPI

  • Active QA User ≥ 90%
  • งานใหม่เข้าระบบ ≥ 80% ภายใน 60 วัน
  • Excel Manual Tracking ลดลง ≥ 60%
  • User Satisfaction ≥ 4/5
  • Support Ticket จากระบบลดลงหลังเดือนที่ 3

14. Go-live, Handover & Operating Model

แผนเปลี่ยนจาก Project ไปเป็นระบบใช้งานจริง

Go-live Sequence

1. Freeze MVP Scope 2. Production Readiness Review 3. Backup & Rollback Drill 4. Migrate Product / User / Test Case 5. Pilot with ProMaxx and MyMaxx2 6. Hypercare 2–4 weeks 7. Review KPI and Critical Issues 8. Roll out remaining Products 9. Switch to BAU Support

BAU Operating Model

  • Product Owner ดูแล Backlog
  • QA Platform Admin ดูแล Master/Permission
  • Dev Team ดูแล Application Defect
  • DevOps ดูแล Infrastructure/Monitoring
  • AI Owner ดูแล Prompt, Model และ Accuracy
  • รายเดือนทบทวน KPI, Risk และ Adoption
ข้อเสนอการเริ่มต้น: ให้ล็อก MVP ที่ Phase 1–7 ก่อน จากนั้น Pilot กับ ProMaxx และ MyMaxx2 เมื่อ Workflow หลักนิ่งแล้วจึงเพิ่ม Toolbox, Integration และ AI เพื่อลดความเสี่ยงด้าน Scope และ Data Model
